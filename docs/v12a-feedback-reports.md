# V12A：反馈 / 报错系统

V12A 的学习主题是：用户反馈和支持流程。

真实产品上线后，开发者最怕的不是“本地有 bug”，而是“用户那里坏了，但我们不知道”。反馈系统就是让用户把问题带着上下文提交给团队。

## 这一版做什么

- 新增 `feedback_reports` 表。
- 登录用户可以提交反馈。
- 反馈会带上当前组织、页面地址、浏览器信息和应用版本。
- 管理员可以查看最近 20 条反馈。
- 普通用户不能查看别人的反馈。

## 关键概念

- `feedback report`：反馈报告。
- `client metadata`：客户端元信息，比如浏览器和页面地址。
- `bug reproduction`：复现问题需要的信息。
- `support workflow`：用户反馈后，团队如何查看和处理。
- `RLS`：数据库层限制谁能提交、谁能查看。

## 为什么重要

没有反馈系统时，用户只能口头说：

```text
它坏了。
```

有反馈系统后，我们至少能看到：

```text
谁提交的
在哪个组织
在哪个页面
什么浏览器
什么时候提交
用户描述了什么
```

这会让修 bug 从猜谜变成调查。

## 你需要做什么

在 Supabase SQL Editor 执行：

```text
docs/v12a-feedback-reports-schema.sql
```

执行成功后：

1. 登录任意云端账号。
2. 在 `反馈 / 报错` 面板提交一条反馈。
3. 用管理员账号登录。
4. 点击 `查看最近反馈`。
5. 确认能看到刚提交的反馈。

## 真实 SaaS 还差什么

V12A 是轻量反馈系统。更完整的生产系统通常还会继续做：

- 反馈状态流转，比如 open / triaged / resolved；
- 管理员回复用户；
- 错误截图或附件；
- 自动捕获 JavaScript 错误；
- 接入 Sentry 这类专业错误监控服务；
- 反馈通知，比如发到邮件、Slack 或企业微信。
